프로젝트 선택

모니터링할 WhaTap 프로젝트를 선택합니다

WhaTap 계정에 연결된 프로젝트가 카드 형태로 표시됩니다. 프로젝트 이름, pcode, 유형(JAVA, NODEJS, PYTHON 등)을 확인하고 클릭하면 모니터링이 시작됩니다.

모니터 — 실시간 모니터링

Scout Agent가 3초 간격으로 WhaTap에서 메트릭을 수집합니다

모니터 대시보드 — 메트릭 수집 시작 직후

데이터 수집 중 — 메트릭 값이 표시되고 이상 감지가 시작된 모습

상단 요약: 활성 이슈, 오늘 수정, 마지막 수정, 활성 에이전트

메트릭 카드: CPU, Memory, TPS, Error Rate, Active Tx, Resp Time — 임계값 초과 시 테두리 색상 변경

Scout 에이전트 로그: 수집된 메트릭 기록과 상태(OK / WARN / ISSUE)

좌측 사이드바: 페이지 네비게이션 + 에이전트 상태 (Scout, Analyzer, Fixer, Deployer, Verifier)

설정 — 연결, AI 분석, 임계값

서비스 연결 상태, AI 분석 제공자, 임계값, 데이터 관리

연결: SessionCast (세션 녹화), WhaTap (모니터링), GitHub (코드 저장소), 일반 (폴링 간격, 동시 실행)

AI 분석: Claude Code, Codex CLI, Gemini CLI, Cursor CLI, Ollama 중 선택

임계값: CPU, 메모리, 에러율, 응답 시간, 디스크, TPS 감소 비율

데이터 관리: 데이터 초기화, 로그아웃 및 전체 초기화

AI 분석 — Claude Code (기본), SessionCast Relay 연결 상태

AI 제공자 변경 — Codex CLI 선택 예시

브라우저를 닫아도 서버가 실행 중이면 분석은 계속됩니다. 데이터는 서버 메모리에 저장되며, 서버 재시작 시 초기화됩니다.

6

실제 배포 vs 시뮬레이션 모드

GitHub 연동 여부에 따라 배포 방식이 달라집니다

실제 배포 모드

GitHub + SessionCast 연동 시

  1. 1. Fixer — AI가 코드 diff를 생성하고 자동 PR 생성
  2. 2. Deployer — Git merge → CI/CD 배포 실행, SessionCast로 세션 녹화
  3. 3. Verifier — 배포 후 메트릭 재측정, PASS/FAIL 자동 판정

필요 조건: GitHub 토큰 + 저장소 설정

시뮬레이션 모드

GitHub 미연동 시 (기본 동작)

  1. 1. Fixer — AI가 권장 조치 목록을 생성 (코드 수정 대신)
  2. 2. Deployer — 시뮬레이션 터미널 UI로 배포 과정 시각화
  3. 3. Verifier — Before/After 메트릭 비교, 수동 조치 대기(PENDING)

별도 설정 불필요 — GitHub 미연동 시 자동 전환

시뮬레이션 모드는 언제 사용되나요?

  • GitHub 토큰이 설정되지 않은 경우 (기본 상태)
  • 이슈 유형이 코드 수정이 아닌 설정 변경이나 인프라 조치가 필요한 경우 (예: RESPONSE_SLOW)
  • Fixer Agent가 autoFixAvailable: false로 판단한 경우

시뮬레이션 모드에서도 다음은 실제로 동작합니다:

  • Scout Agent의 메트릭 수집 — WhaTap API에서 실제 메트릭을 폴링
  • Analyzer Agent의 AI 분석 — SessionCast Relay를 통해 실제 LLM(Claude Code 등)이 분석
  • Verifier Agent의 메트릭 재확인 — 배포 후 WhaTap에서 실제 메트릭을 다시 측정

[상태] GitHub 연동 없음 — 시뮬레이션 모드로 진행

[권장 조치]

1) HikariCP maximum-pool-size: 20→50 증가

2) server.tomcat.threads.max: 200→400

3) 활성 트랜잭션이 높으면 slow query 점검 필요

[시뮬레이션 배포 로그]

$ kubectl set env deployment/api-server --dry-run=client

→ 설정 변경이 적용되면 pod 재시작이 필요합니다

[결론] 수동 조치 후 Verifier가 메트릭을 재확인합니다

시뮬레이션 터미널이 보여주는 내용:

  • [상태] — GitHub 연동 여부, 시뮬레이션 모드 안내
  • [권장 조치] — Fixer Agent가 분석한 구체적인 조치 목록
  • [시뮬레이션 배포 로그] — 실제 배포 시 실행될 명령어 시뮬레이션 (kubectl, docker 등)
  • [결론] — 다음 단계 안내 (수동 조치 → Verifier 메트릭 재확인)

Verifier Agent — Before/After 비교

시뮬레이션 모드에서의 Verifier 동작:

  1. 배포(시뮬레이션) 후 10초 대기
  2. WhaTap에서 After 메트릭을 재측정
  3. Before 값과 After 값을 비교하여 판정
PASS
해결됨
메트릭이 임계값 이내로 회복
IMPROVING
개선 중
메트릭 개선되었으나 아직 임계값 초과
PENDING
대기 중
수동 조치 적용 대기

시뮬레이션 모드에서 PENDING은 정상입니다. 권장 조치를 수동으로 적용한 후, 다음 Scout 폴링 주기에서 자동으로 개선 여부가 확인됩니다.

8

CPU_HIGH 파이프라인 상세

CPU 사용률 임계값 초과 — 자동 감지 → AI 분석 → 수정안 → 시뮬레이션 배포 → 검증

파이프라인 상단에 이슈 타입, 현재 값(2.23), 임계값(1), 5단계 진행 상태 바가 표시됩니다.

Scout Agent: whatap mxql --pcode ? --category app_counter --field cpu 명령으로 메트릭을 수집합니다.

Analyzer Agent — AI 근본 원인 분석 (한국어)

SessionCast Relay → Claude Code를 통한 실제 LLM 분석 결과:

  • 근본 원인: "CPU 사용률이 임계값(1.0)의 2.2배를 초과했으며, 응답시간이 1885ms로 높게 측정되어 활성 트랜잭션(8개) 처리 지연이 CPU 부하를 유발하고 있는 것으로 판단됨"
  • 상관관계: ResponseTime:1885ms(정상 대비 높음), ActiveTx:8(처리 중인 트랜잭션 다수), TPS:1(낮은 처리량 대비 높은 응답시간)
  • 신뢰도: 62%

Fixer → Deployer → Verifier

Fixer: HikariCP 풀, Tomcat 스레드 조정 권장 조치

Deployer: 시뮬레이션 배포 터미널 + kubectl 명령어

Verifier Agent가 Before(2.2%) → After(2.2%) 메트릭을 비교합니다. 시뮬레이션 모드에서는 수동 조치를 적용하기 전까지 PENDING 상태입니다.

9

RESPONSE_SLOW 파이프라인 상세

응답 시간 지연 감지 — AI 원인 분석 및 권장 조치 제시

Analyzer Agent — AI 근본 원인 분석 (한국어)

LLM 분석 결과:

  • 근본 원인: "낮은 TPS(1)와 낮은 CPU/메모리 사용률에도 불구하고 응답시간이 3277ms로 급등한 것은 외부 서비스 호출 대기, DB 쿼리 지연, 또는 I/O 블로킹에 의한 처리 병목이 원인"
  • 상관관계: TPS:1 (매우 낮은 처리량), CPU:2.1% (리소스 여유 있음), ActiveTx:6 (소수 트랜잭션이 장시간 점유), DBPool:0.0% (DB 커넥션 풀 미사용 또는 측정 오류)
  • 권장 조치: Slow query 로그 확인, DB 커넥션 풀 사용량 확인, 캐시 적용 가능한 API 점검, N+1 쿼리 패턴 확인

Fixer: DB 쿼리 및 연결 풀 최적화 권장 조치

Deployer: 시뮬레이션 배포 + 권장 조치 기록

RESPONSE_SLOW는 수동 조치가 필요한 유형입니다. Fixer Agent가 코드 수정 대신 권장 조치 목록을 제시하며, Verifier Agent가 수동 조치 적용 후 메트릭 개선 여부를 Before/After로 비교합니다.

10

진행 중(IN_PROGRESS) 파이프라인

실시간으로 분석이 진행되는 파이프라인의 모습

진행 중 상태의 특징:

  • 현재 단계가 파란색 펄스 애니메이션으로 표시됩니다
  • 이후 단계는 회색(대기)으로 표시됩니다
  • 사이드바에서 해당 에이전트가 "N 작업"으로 표시됩니다
  • Scout Agent가 새로운 이슈를 감지할 때마다 자동으로 새 파이프라인이 생성됩니다

Analyzer 단계 진행 중 — AI가 분석 중인 상태

Fixer 단계 — 수정안/권장 조치 생성 중

1. Scout Agent — WhaTap CLI

WhaTap API에서 3초 간격으로 메트릭(CPU, 메모리, TPS, 에러율, 응답시간, DB풀)을 폴링합니다. 임계값 초과 시 자동으로 파이프라인을 생성합니다.

2. Analyzer Agent — SessionCast Relay → LLM

SessionCast Relay를 통해 로컬 AI CLI(Claude Code 등)에 한국어로 분석을 요청합니다. 근본 원인, 상관 메트릭, 신뢰도를 산출합니다. SessionCast 미연결 시 규칙 기반 폴백 분석을 수행합니다.

3. Fixer Agent — GitHub / AI

분석 결과를 바탕으로 코드 수정(diff) 또는 권장 조치를 생성합니다. GitHub 연동 시 자동 PR 생성, 미연동 시 권장 조치 목록을 제시합니다.

4. Deployer Agent — Git/CI/CD 또는 시뮬레이션

GitHub 연동 시 실제 배포(merge → deploy), 미연동 시 시뮬레이션 모드로 동작합니다. 시뮬레이션에서는 배포 과정을 터미널 UI로 시각화하며, 권장 조치를 기록합니다.

5. Verifier Agent — WhaTap CLI

배포(또는 시뮬레이션) 후 WhaTap 메트릭을 재확인합니다. Before/After 비교로 해결됨(PASS), 개선 중(IMPROVING), 대기 중(PENDING), 임계값 초과 지속(FAIL)을 판정합니다.
